# Accessibility Review — badwers-notify

## Overall verdict

The spec has strong a11y intent but contains several WCAG 2.1 AA failures in color contrast (success-light at 3.13:1, attention-light at 2.85:1), a critical touch-target violation (20×20px close button), and no `prefers-reduced-motion` support. The ARIA strategy has a conflict (`role="alert"` + `aria-live="polite"` on the same element) and the placement of `role="alert"` on the container rather than individual cards will miss announcements in some screen readers. Keyboard focus management after closing a notification is unhandled. These findings need resolution before release.

## Findings

- **critical** success-light contrast: #fff on #28a745 (3.13:1) fails WCAG AA for normal-size text — title (15px semibold ≈ 11.25pt) and message (14px, 10.5pt) are both below the large-text threshold. (EXPERIENCE.md:129, DESIGN.md:8-9). *Fix: darken `success-bg` to ≥ #1e8a3a to reach 4.5:1.*
- **critical** attention-light contrast: #fff on #e67e22 (2.85:1) fails WCAG AA entirely — does not meet even the 3:1 large-text threshold. (EXPERIENCE.md:129, DESIGN.md:10-11). *Fix: darken `attention-bg` to ≥ #b85c00 (4.60:1, already passes in dark theme).*
- **critical** close-button touch target: 20×20px fails WCAG 2.2 AA (24×24px minimum) and AAA (44×44px). (DESIGN.md:84-86). *Fix: increase `notify-close` to 24×24px (AA) or 44×44px (AAA); use `padding` to enlarge hit area without enlarging the visible icon.*
- **high** `role="alert"` placed on Stack Container instead of each notification card — screen readers may not announce dynamically inserted content when `role="alert"` is on the persistent ancestor. (EXPERIENCE.md:125). *Fix: set `role="alert"` on each individual notification element when it is inserted into the DOM.*
- **high** conflicting ARIA — `role="alert"` (implicit `aria-live="assertive"`) and explicit `aria-live="polite"` on the same element creates indeterminate behavior. (EXPERIENCE.md:125-126). *Fix: choose one — use `aria-live="polite"` on the container OR `role="alert"` on individual cards, not both.*
- **high** no `prefers-reduced-motion` support — 300ms slide+fade animations run regardless of user motion preferences. (DESIGN.md:66, EXPERIENCE.md:94). *Fix: wrap transitions in `@media (prefers-reduced-motion: reduce)` with zero-duration or instant-opacity alternative.*
- **high** focus lost on keyboard close — when close button is focused and the notification is removed, focus falls to `<body>` with no management. (EXPERIENCE.md:99-104). *Fix: move focus to the next notification's close button, or to the container's first focusable element, or to the triggering element via `onClose` callback parameter.*
- **high** close-button resting opacity reduces effective contrast — `opacity: 0.6` on the close button blends its text with the background, lowering contrast below the calculated value. (DESIGN.md:88). *Fix: use a specific accessible color instead of opacity for the resting state (e.g., `color: #666` on info-light, `color: rgba(255,255,255,0.85)` on colored types).*
- **medium** no visible focus indicator on close button — keyboard users cannot see where focus is. (key-screens.html:86-99). *Fix: add `:focus-visible { outline: 2px solid currentColor; outline-offset: 2px; }` or a `box-shadow` ring.*
- **medium** `aria-label` on close button hardcoded to Russian — not configurable or localizable. (EXPERIENCE.md:127). *Fix: accept `closeAriaLabel` as an option with `"Закрыть уведомление"` as default, or use a public API for locale overrides.*
- **low** `aria-live="polite"` is appropriate for non-critical notifications BUT `warning`/`attention` types may warrant `aria-live="assertive"` due to urgency. (EXPERIENCE.md:126). *Fix: consider mapping `warning` → `assertive`, others → `polite`.*
- **low** success/attention/warning icons use #fff strokes on colored backgrounds where small decorative graphics may fall below 3:1 contrast — acceptable for purely decorative `aria-hidden="true"` icons. (key-screens.html:142, 150, 171). *No fix needed if `aria-hidden="true"` is applied; verify in implementation.*
- **info** info-light contrast (15.91:1), warning-light (4.53:1), and all dark-theme pairs pass WCAG AA — the spec's claim in EXPERIENCE.md:129 that "all types (except info-light) have contrast > 4.5:1" is incorrect for success-light and attention-light.
